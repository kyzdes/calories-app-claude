# CalorieMate — PRD (Product Requirements Document)

> **Version:** 1.0 (MVP)  
> **Дата:** 21.02.2026  
> **Платформа:** iOS 17+ (SwiftUI)  
> **Рынок:** Россия / СНГ  
> **Монетизация:** Бесплатно (MVP)

---

## 1. Executive Summary

CalorieMate — мобильное iOS-приложение для контроля калорий и питания, созданное как альтернатива FatSecret, MyFitnessPal и Weight Watchers для российского рынка. Ключевое отличие — минималистичный интерфейс и максимально быстрый процесс логирования еды, который решает две главные проблемы существующих решений: перегруженность интерфейса и сложность внесения данных.

### Product Vision

> *«Залогировать еду за 10 секунд, понять свой прогресс за 3 секунды. Ничего лишнего.»*

### Ключевые метрики успеха (North Star)

| Метрика | Цель (3 мес.) | Цель (6 мес.) |
|---------|---------------|----------------|
| Среднее время логирования 1 приёма пищи | ≤ 15 сек | ≤ 10 сек |
| D7 Retention | ≥ 35% | ≥ 45% |
| DAU/MAU | ≥ 25% | ≥ 35% |
| Логирований / DAU / день | ≥ 2.5 | ≥ 3.0 |
| App Store Rating | ≥ 4.5 | ≥ 4.7 |

## 2. Problem Statement

### 2.1 Боли пользователей

Анализ отзывов в App Store и пользовательских исследований выявил два ключевых барьера, из-за которых пользователи бросают трекеры калорий:

| Проблема | Описание | Наше решение |
|----------|----------|--------------|
| **Перегруженный UI** | Слишком много экранов, функций, рекламы и upsell-элементов. Пользователь теряется в интерфейсе и чувствует когнитивную перегрузку. | 4 основных экрана, ноль рекламы, фокус на одной задаче на экране |
| **Долгое логирование** | Добавление одного продукта занимает 30–60 сек, много тапов и переходов. Пользователи бросают логирование через 1–2 недели. | 1-tap из недавних, мгновенный сканер, inline-редактирование порций |

### 2.2 Специфика рынка РФ/СНГ

- Глобальные базы продуктов (Nutritionix, USDA) плохо покрывают российские продукты и бренды
- FatSecret имеет русскую локализацию, но устаревший UI и медленный UX
- MyFitnessPal частично недоступен или ограничен в РФ, оплата премиума затруднена
- Российские пользователи привыкли к системе БЖУ (белки/жиры/углеводы) и метрической системе (граммы, кг)

## 3. Целевая аудитория

### 3.1 Первичный сегмент

Мужчины и женщины 22–40 лет, Россия и СНГ. Хотят контролировать питание для похудения, набора массы или поддержания формы. Ценят скорость и простоту. Устали от перегруженных приложений.

### 3.2 User Personas

| Persona | Цель | Боль | Сценарий |
|---------|------|------|----------|
| **Лиза, 28** | Похудеть на 8 кг к лету | Бросила MFP через неделю — слишком долго вносить еду | Логирует завтрак/обед/ужин, следит за дефицитом ккал |
| **Дима, 32** | Набрать мышечную массу | FatSecret не находит российское спортпит | Сканирует протеин и еду, следит за БЖУ и весом |
| **Марина, 35** | Поддерживать вес после диеты | Нужен простой дневник без сложных функций | Быстро логирует из недавних, проверяет графики |

## 6. Feature Specification (MVP)

### 6.1 Onboarding

Цель: собрать минимум данных для расчёта дневной нормы ккал. Максимум 5 шагов, без регистрации на старте.

| Шаг | Содержание | UI-элемент |
|-----|------------|------------|
| 1 | Цель: похудеть / набрать / поддерживать | 3 карточки с иконками |
| 2 | Пол: мужской / женский | Segmented control |
| 3 | Возраст, рост, текущий вес | Number pickers с крупными цифрами |
| 4 | Уровень активности (4 варианта) | Список с описаниями |
| 5 | Расчёт нормы + целевой вес (опц.) | Карточка-результат + кнопка «Начать» |

**Формула расчёта:** Mifflin-St Jeor (BMR) × коэффициент активности (TDEE) ± дефицит/профицит (по умолчанию 20%). Распределение БЖУ по умолчанию: 30/25/45 (похудение), 30/30/40 (набор), 25/30/45 (поддержание). Пользователь может изменить в профиле.

### 6.2 Дашборд дня (таб «Сегодня»)

Главный экран приложения. Пользователь видит всю ключевую информацию без скролла.

#### Элементы экрана (сверху вниз)

1. **Круговой прогресс-бар калорий.** Центр: «съедено / цель». Цвет: зелёный → жёлтый (>90%) → красный (>100%). Под кругом — «осталось X ккал».
2. **БЖУ-бары.** Три горизонтальных прогресс-бара: Белки (синий), Жиры (оранжевый), Углеводы (фиолетовый). Подпись: «X / Y г».
3. **Список приёмов пищи.** 4 секции: Завтрак, Обед, Ужин, Перекус. Каждая секция показывает сумму ккал и кнопку «+». Swipe-to-delete на продуктах.
4. **Навигация по дням.** Swipe влево/вправо для переключения между днями. Тап на дату — calendar picker.

### 6.3 Логирование еды

**Это ключевой flow приложения** — именно здесь конкуренты теряют пользователей. Целевое время: ≤ 10 секунд на добавление одного продукта.

#### A. Три способа добавления

| Способ | Flow | Тапов |
|--------|------|-------|
| **Быстрое добавление** | Нажать «+» → Видит «Недавние» и «Частые» сразу → Тап на продукт → Подтвердить порцию (предзаполнена) | 2–3 |
| **Поиск** | Нажать «+» → Начать ввод в поиске → Выбрать продукт → Указать порцию → Сохранить | 4–5 |
| **Сканер штрих-кода** | Нажать «+» → Тап на иконку сканера → Навести камеру → Авто-распознавание → Подтвердить | 3–4 |

#### B. Экран добавления продукта

Открывается как sheet (не full-screen push), чтобы пользователь видел контекст дня. Содержит:

- **Поисковая строка** — автофокус, результаты после 2 символов, debounce 300ms
- **Кнопка сканера** — справа от поисковой строки
- **Секция «Недавние»** — последние 20 уникальных продуктов с последней порцией
- **Секция «Частые»** — топ-10 по частоте за последние 30 дней
- **Кнопка «Создать продукт» (UGC)** — ручной ввод названия, ккал, БЖУ на 100г

#### C. Экран порции

После выбора продукта открывается compact sheet:

- Название продукта и калорийность на 100г
- **Stepper порции** — предустановки: 50г, 100г, 150г, 200г + свободный ввод граммов
- Динамический пересчёт ккал/БЖУ при изменении порции
- Выбор приёма пищи (если не был выбран ранее)
- **Кнопка «Добавить»** — сохраняет и закрывает sheet с haptic feedback

### 6.4 Сканер штрих-кодов

| Параметр | Спецификация |
|----------|-------------|
| Фреймворк | AVFoundation (VNDetectBarcodesRequest) |
| Форматы | EAN-13, EAN-8, UPC-A, UPC-E, Code128 |
| Поведение | Авто-скан без кнопки + haptic при успехе |
| Найден в базе | Переход на экран порции |
| Не найден | Предложить создать продукт (UGC) с предзаполненным штрих-кодом |
| Оффлайн | Кэш последних 500 отсканированных продуктов |

### 6.5 Трекинг тела (таб «Тело»)

#### Вес

- Быстрый ввод веса с number pad (одно поле, кг с десятыми)
- График веса за период (7д / 30д / 90д / всё) с линией тренда
- Отображение дельты: «−2.3 кг за 30 дней»
- Синхронизация с HealthKit (двусторонняя)

#### Замеры тела (опционально)

- Талия, бёдра, грудь, бицепс (см)
- Фото прогресса (опционально, хранение локально)

### 6.6 Прогресс (таб «Прогресс»)

Исторические данные в виде графиков. Периоды: неделя / месяц / 3 месяца / всё время.

- **Калории по дням** — столбчатый график с линией цели
- **БЖУ по дням** — stacked bar chart
- **Вес** — линейный график с трендом
- **Средние значения** — карточки сверху: средние ккал, БЖУ, соблюдение нормы

**Библиотека графиков:** Swift Charts (iOS 16+). Все графики интерактивные — тап на точку показывает tooltip с деталями.

### 6.7 Профиль

- Цель и дневная норма ккал (редактируемая)
- Распределение БЖУ (редактируемое, % или граммы)
- Личные данные (возраст, рост, пол, активность)
- Настройки HealthKit
- Экспорт данных (CSV)
- О приложении / Обратная связь

## 7. База продуктов

### 7.1 Архитектура

Гибридная модель: Open Food Facts (OFF) как основная база + UGC (пользовательский контент) для дополнения.

| Источник | Описание | Приоритет в поиске |
|----------|----------|--------------------|
| **Open Food Facts API** | ~3M продуктов, хорошее покрытие RU-сегмента. Бесплатный API, лицензия ODbL. | Основной |
| **UGC-база** | Пользователи добавляют продукты, которых нет в OFF. Хранятся на нашем бэкенде. | Дополнительный |
| **Локальный кэш** | Недавние и частые продукты хранятся на устройстве (Core Data). | Первый (оффлайн) |

### 7.2 Поиск

Порядок поиска: локальный кэш → UGC-база → Open Food Facts API. Результаты объединяются и ранжируются по релевантности + популярности среди пользователей.

### 7.3 Модель данных продукта

| Поле | Тип | Обязательное |
|------|-----|-------------|
| id | UUID | Да |
| name | String | Да |
| barcode | String? | Нет |
| calories_per_100g | Double | Да |
| protein_per_100g | Double | Да |
| fat_per_100g | Double | Да |
| carbs_per_100g | Double | Да |
| source | Enum (off/ugc/manual) | Да |
| serving_sizes | [ServingSize] | Нет |
| brand | String? | Нет |

## 8. Интеграция HealthKit

Интеграция с Apple Health позволяет синхронизировать данные и не дублировать ввод.

| Тип данных | Чтение | Запись | Назначение |
|------------|--------|--------|------------|
| bodyMass | ✓ | ✓ | Синхронизация веса с Apple Health / весами |
| height | ✓ | — | Расчёт BMR |
| biologicalSex | ✓ | — | Расчёт BMR |
| dateOfBirth | ✓ | — | Расчёт BMR |
| dietaryEnergyConsumed | — | ✓ | Экспорт калорий в Health |
| activeEnergyBurned | ✓ | — | Коррекция TDEE по активности (v2) |

**Важно:** HealthKit опционален. Приложение полностью работает без него. Запрос разрешений — после onboarding, с объяснением зачем это нужно.

## 9. Техническая архитектура

### 9.1 Стек

| Слой | Технология |
|------|-----------|
| Язык | Swift 5.9+ |
| UI | SwiftUI + Swift Charts |
| Min iOS | 17.0 |
| Архитектура | MVVM + SwiftData / Core Data |
| Локальное хранение | SwiftData (основные данные) + UserDefaults (настройки) |
| Сеть | URLSession + async/await |
| Сканер | AVFoundation (VNDetectBarcodesRequest) |
| Здоровье | HealthKit |
| Бэкенд (MVP) | Firebase (Auth + Firestore + Cloud Functions) |
| CI/CD | Xcode Cloud / Fastlane |

### 9.2 Архитектурные решения

- **Offline-first:** все данные питания хранятся локально, синхронизация с облаком в фоне
- **API-first поиск:** локальный кэш → сервер (UGC) → Open Food Facts. Результаты мержатся в один список
- **Нет обязательной регистрации:** приложение работает локально. Регистрация — для UGC-контента, бэкапа и синхронизации между устройствами

## 10. Ключевые UX Flows

### 10.1 First-Time User Experience

App Launch → Welcome screen (1 экран, без карусели) → Onboarding (5 шагов) → Результат расчёта → HealthKit запрос (опционально) → Таб «Сегодня». Время до первого логирования: ≤ 90 секунд.

### 10.2 Daily Logging Flow (основной)

Открыть приложение → Видит дашборд с остатком ккал → Тап «+» у нужного приёма пищи (FAB или секция) → Выбрать из недавних / поиск / сканер → Подтвердить порцию → Вернуться на дашборд (обновлён). Цель: ≤ 10 секунд на продукт из недавних.

### 10.3 Weight Logging Flow

Таб «Тело» → Тап «+» или на текущее значение → Number pad с предзаполненным последним весом → Сохранить → График обновлён + haptic. Цель: ≤ 5 секунд.

## 15. Критерии успеха MVP

Продукт считается успешным после 3 месяцев после запуска, если:

- **D7 Retention ≥ 35%** — пользователи возвращаются через неделю
- **Среднее время логирования ≤ 15 сек** — ключевое конкурентное преимущество
- **App Store Rating ≥ 4.5** — пользователи довольны продуктом
- **NPS ≥ 40** — готовы рекомендовать
- **Органические установки ≥ 70%** — продукт растёт органически (сарафанное радио + ASO)
